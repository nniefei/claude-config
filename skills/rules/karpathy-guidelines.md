# Karpathy 准则

> **自动加载触发**：进入标准/静默模式时，本文件自动加载到执行上下文（详见 `rule-loading.md`）。
>
> 基于 Andrej Karpathy 对 LLM 编码陷阱的观察，总结出的行为准则，旨在减少常见的编码错误。

**权衡：** 这些准则偏向谨慎，可能牺牲速度。对于琐碎任务，请自行判断。

## 1. 先思考，后编码

**不要假设。不要隐藏困惑。把权衡摆到台面上。**

在实现之前：
- 明确陈述你的假设。如有不确定，立刻提问。
- 如果存在多种理解方式，把它们列出来——不要私下默认选择某一种。
- 如果有更简单的做法，就直说。该反对时就反对。
- 如果有不清楚的地方，停下来。说出哪里令人困惑，然后提问。

## 2. 简单至上

**用最少的代码解决问题。不做任何投机性设计。**

- 不要添加未要求的功能。
- 不要为一次性使用的代码构建抽象。
- 不要添加未被要求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景编写错误处理。
- 如果你写了 200 行代码而本可以用 50 行实现，就重写它。

问问自己：“资深工程师会认为这过于复杂吗？” 如果是，就简化。

## 3. 手术式修改

**只改动必须改的地方。只清理自己造成的乱摊子。**

在修改现有代码时：
- 不要“顺便改进”相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你个人偏好不同。
- 如果注意到无关的废弃代码，可以提一句——但不要删掉。

当你的改动产生废弃引用时：
- 删除因 **你的改动** 而不再使用的导入、变量或函数。
- 除非被要求，否则不要删除已有的废弃代码。

检验标准：每行被改动的代码都必须能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环执行直至验证通过。**

将任务转化为可验证的目标：
- “增加校验” → “为非法输入编写测试，然后让测试通过”
- “修复 bug” → “编写一个能复现该 bug 的测试，然后让测试通过”
- “重构 X” → “确保重构前后测试均通过”

对于多步骤任务，简述计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```
强有的成功标准让你能够独立循环推进。弱标准（如“把它搞定”）则需要不断请求澄清。

## 5. 失败模式自查

**识别并避免这几类反复出现的翻车套路。一旦发现自己陷入，停下，而不是硬着头皮继续。**

- **厨房水槽**：顺手重构了半个代码库。
- **错误抽象**：只用了两次就急着抽象。
- **乐观路径**：只处理了快乐路径，却忽略了 500 错误等异常分支。
- **失控重构**：一次修复级联扩散到多个文件。

检验标准：发现 diff 在不该大的地方变大，立刻停下来收缩。
